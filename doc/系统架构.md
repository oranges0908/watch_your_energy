## 三、数据库 Schema

### 设计原则
- 时间字段统一使用 Unix 毫秒整数
- 软删除（`status` 字段），不物理删除数据
- `app_state` 单行表管理全局运行状态

### 表结构

#### `projects`
```sql
CREATE TABLE projects (
    id          TEXT    PRIMARY KEY,             -- UUID
    title       TEXT    NOT NULL,                -- "优化简历"
    status      TEXT    NOT NULL DEFAULT 'active', -- active | completed | archived
    sort_order  INTEGER NOT NULL DEFAULT 0,       -- 侧边栏排序（recency）
    created_at  INTEGER NOT NULL,
    updated_at  INTEGER NOT NULL
);
```
> 约束：应用层强制 `active` 数量 ≤ 3

#### `blocks`（结构块）
```sql
CREATE TABLE blocks (
    id           TEXT    PRIMARY KEY,
    project_id   TEXT    NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    title        TEXT    NOT NULL,               -- "项目1" | "项目2" | "项目3" | "总结"
    position     INTEGER NOT NULL,               -- 块内显示顺序
    status       TEXT    NOT NULL DEFAULT 'not_started',
                         -- not_started | in_progress | near_complete | completed
    progress_pct INTEGER NOT NULL DEFAULT 0,     -- 0–100
    created_at   INTEGER NOT NULL,
    updated_at   INTEGER NOT NULL
);
```

#### `steps`
```sql
CREATE TABLE steps (
    id            TEXT    PRIMARY KEY,
    project_id    TEXT    NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    block_id      TEXT    REFERENCES blocks(id),
    description   TEXT    NOT NULL,    -- "写项目2的第一句话（描述你做了什么）"
    pattern       TEXT    NOT NULL,    -- Light|Continuation|Completion|Decomposition|Recovery|Refinement
    step_type     TEXT    NOT NULL,    -- Bootstrap|Push|Wrap|Transfer
    estimated_min INTEGER NOT NULL,    -- 5–20
    energy_level  TEXT    NOT NULL DEFAULT 'normal',  -- normal | low
    trigger       TEXT    NOT NULL,    -- auto|skip|stuck|low_energy|resume
    status        TEXT    NOT NULL DEFAULT 'pending',
                          -- pending | active | completed | skipped | stuck
    created_at    INTEGER NOT NULL,
    completed_at  INTEGER,             -- NULL 直到完成
    session_id    TEXT    NOT NULL
);
```

#### `step_rejections`（多样性追踪）
```sql
CREATE TABLE step_rejections (
    id          TEXT    PRIMARY KEY,
    project_id  TEXT    NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    step_id     TEXT    NOT NULL REFERENCES steps(id),
    verb        TEXT    NOT NULL,      -- 从 description 提取的动词
    object_text TEXT    NOT NULL,      -- 从 description 提取的对象
    pattern     TEXT    NOT NULL,
    rejected_at INTEGER NOT NULL
);
```
> 每个 project 只保留最近 3 条，由 `diversity_service.py` 维护

#### `sessions`
```sql
CREATE TABLE sessions (
    id          TEXT    PRIMARY KEY,
    started_at  INTEGER NOT NULL,
    ended_at    INTEGER,               -- NULL = 当前 session
    energy_mode TEXT    NOT NULL DEFAULT 'normal'
);
```

#### `app_state`（单行全局状态）
```sql
CREATE TABLE app_state (
    id                  INTEGER PRIMARY KEY DEFAULT 1,
    active_project_id   TEXT    REFERENCES projects(id),
    active_session_id   TEXT    REFERENCES sessions(id),
    energy_mode         TEXT    NOT NULL DEFAULT 'normal',
    onboarding_complete INTEGER NOT NULL DEFAULT 0
);
```

### Block 进度增量常量（`config.py`）

| Step类型 | 能量模式 | 进度增量 |
|---|---|---|
| Bootstrap | normal | +20% |
| Push | normal | +15% |
| Wrap | normal | +10% |
| Transfer | normal | 0%（触发转移到下一块逻辑） |
| 任意 | low | +5% |

项目总进度 = 所有 blocks 的 `progress_pct` 均值

## 五、API 端点

### `/state` — 全局状态

| Method | Path | 说明 |
|---|---|---|
| GET | `/state` | 返回当前 step、project、energy_mode、session_id |
| PATCH | `/state/energy` | 切换能量模式 `{mode: "normal"\|"low"}` → 丢弃当前step → 重新生成 |
| POST | `/state/session` | App打开时调用：创建新session、检测中断（返回 trigger） |

### `/projects` — 项目管理

| Method | Path | 说明 |
|---|---|---|
| GET | `/projects` | 列出所有 active 项目（≤3），按 `updated_at` 降序 |
| POST | `/projects` | 创建：`{title, completed_block_ids[]}` → 生成4块 → 生成首个Step |
| GET | `/projects/{id}` | 项目详情 + blocks状态（供进展页使用） |
| PATCH | `/projects/{id}/active` | 设为活跃项目 → 返回该项目的当前Step |
| DELETE | `/projects/{id}` | 软归档（status → archived） |

### `/steps` — 步骤操作

| Method | Path | 说明 |
|---|---|---|
| GET | `/steps/current` | 当前 pending/active step |
| POST | `/steps/start` | 点击「开始」→ status = active |
| POST | `/steps/complete` | 完成 → 更新block进度 → 生成下一个Step |
| POST | `/steps/skip` | 「换一个」→ 记录rejection → 生成新Step（多样性约束） |
| POST | `/steps/stuck` | 「我卡住了」→ 记录stuck → 生成同block的Decomposition Step |

### 统一响应体（所有变更操作）

```json
{
  "step": {
    "id": "uuid",
    "description": "把项目1的标题改成一行以内",
    "estimated_min": 10,
    "pattern": "Refinement",
    "block_title": "项目1",
    "energy_level": "normal"
  },
  "project": {
    "id": "uuid",
    "title": "优化简历",
    "progress_pct": 65
  },
  "feedback_message": "项目1已推进"
}
```
> `feedback_message` 仅 `/steps/complete` 时非 null

## 六、LLM Agent 设计

### 核心设计原则

Agent 是一个**受约束的决策引擎**，不是自由对话：
- Pattern 选择由**确定性逻辑**完成，不经过 LLM
- LLM 只负责**创意内容生成**（Step描述文字）
- 所有 DB 读写通过**工具调用**完成，Agent不直接访问DB

### Agent 循环（`agent.py`）

```
输入：trigger, project_id, energy_mode, [block_id]

1. select_pattern(trigger, recent_rejections)  ← 确定性，非LLM
2. tool call: get_project_context(project_id)
3. tool call: select_target_block(project_id, pattern, energy_mode)
4. tool call: get_recent_rejections(project_id, limit=3)
5. tool call: generate_step_candidate(block, pattern, energy_mode, avoid_list)
   └── 内部调用 LLM（唯一调用LLM的工具）
6. tool call: validate_step_format(description)
   └── 失败 → 带错误信息重试，最多2次
   └── 2次失败 → fallback到硬编码模板库
7. tool call: save_generated_step(...)

输出：StepOutput { description, estimated_min, pattern, step_type, block_id }
```

### Pattern 选择逻辑

```python
def select_pattern(trigger, energy_mode, recent_rejections):
    # 连续2次同pattern被拒 → 强制轮换
    if consecutive_same_pattern_rejections(recent_rejections) >= 2:
        return rotate_from(recent_rejections[-1].pattern)

    priority = {
        'low_energy': 'Light',         # 最高优先，无条件
        'stuck':      'Decomposition',
        'resume':     'Recovery',
        'skip':       rotate_from_last(recent_rejections),
        'complete':   'Continuation',
        'auto':       infer_from_block_state(project_context),
    }
    return priority.get(trigger, 'Refinement')
```

`infer_from_block_state`：有 `not_started` 块 → Completion；当前块 `near_complete` → Completion；否则 → Refinement

### 工具清单（`tools.py`）

| 工具 | 职责 |
|---|---|
| `get_project_context` | 读取所有blocks状态、近5条完成step、上次session时间 |
| `select_target_block` | 确定性选块规则：not_started→Bootstrap, in_progress→Push, near_complete→Wrap, completed→Transfer |
| `get_recent_rejections` | 返回最近3条被拒step的 verb + object（多样性约束输入） |
| `generate_step_candidate` | 调用LLM，返回 description + estimated_min |
| `validate_step_format` | 格式校验 + 提取verb/object；失败返回错误描述 |
| `save_generated_step` | 持久化到 steps 表，返回 step_id |

### Step 格式验证器（`validators.py`）

**Pass 1 — 黑名单动词检查**
拒绝以下列词语开头：`["优化", "学习", "思考", "修改", "了解", "研究", "考虑", "整理"]`

**Pass 2 — 结构检查**
- 动词部分：首1–4个汉字
- 对象部分：跟随的名词短语 ≥ 4字
- 总长度：8–30字

失败时返回 `{ valid: false, error: "具体原因" }` 给 Agent 重试。
2次失败后从 fallback 模板库取step（按 block_status + step_type + energy_mode 索引），**保证永不为空**。

### `generate_step_candidate` 系统 Prompt

```
你是一个行动步骤生成器。输出一条中文行动描述。

格式规则（严格遵守）：
[动词] + [具体对象] + [可选：范围限定]

合法示例：
- 写项目2的第一句话（描述你做了什么）
- 把"技能"一栏的三个词改成动词开头
- 给总结段加一个结尾句

非法示例（禁止生成）：
- 优化简历（无具体对象）
- 学习写作技巧（学习类，无可完成标准）

当前上下文：
- 结构块：{block_title}（状态：{block_status}）
- 模式：{pattern}
- 能量状态：{energy_mode}
- 需要回避的动词+对象组合：{avoid_list}

输出格式：
{ "description": "...", "estimated_min": 5-20 }
```

## 八、三条关键数据流

### A. 完成 Step → 生成下一个

```
Flutter: tap "完成"
→ AppStateNotifier.onComplete()
→ POST /steps/complete { step_id }

Backend: StepRouter.complete()
→ step_repo.mark_completed(step_id)
→ block_repo.advance_progress(block_id, +delta)
   └── progress_pct >= 90% → status = near_complete
   └── progress_pct >= 100% → status = completed
→ project_repo.update_progress(project_id)   [avg of blocks]
→ step_service.generate_next(project_id, trigger='complete')
   └── Agent → Continuation pattern → 新Step

→ 返回 { step, project, feedback_message: "项目1已推进" }

Flutter:
→ state = AsyncData(新AppState)
→ StepCard AnimatedSwitcher: 旧Step淡出 → 新Step淡入
→ ProjectHeader 进度条动画: 60% → 65%
→ CompletionFlash: "项目1已推进" 显示2秒后自动消失
```

### B. 卡住 → Decomposition Step

```
Flutter: tap "我卡住了"
→ POST /steps/stuck { step_id }

Backend:
→ step_repo.mark_status(step_id, 'stuck')    [不更新进度]
→ diversity_service.record_stuck(step_id)
→ step_service.generate_next(project_id, trigger='stuck', block_id=同一块)
   └── Agent → Decomposition pattern
   └── select_target_block 强制返回相同 block_id
   └── Prompt: "将当前任务拆解为最小可执行子动作"

→ 返回 { step: 更简单的step, project: 进度不变, feedback_message: null }

Flutter:
→ StepCard 换内容（更简单的描述）
→ 无进度条动画
```

### C. 切换低能量模式 → 轻量 Step

```
Flutter: tap "🌙 低能量"
→ PATCH /state/energy { mode: 'low' }

Backend:
→ state_repo.update_energy_mode('low')
→ step_repo.mark_status(current_pending_step_id, 'skipped')  [丢弃当前step]
→ step_service.generate_next(project_id, trigger='low_energy')
   └── Agent → Light pattern（最高优先，无条件）
   └── Prompt: "5分钟内完成，类型限：回顾型/微调型/确认型/整理型，必须推动结构"
   └── estimated_min 约束为 ≤ 5

→ 返回 { step: 轻量step, project: 进度不变, feedback_message: null }

Flutter:
→ EnergyToggle: 低能量chip高亮
→ StepCard: 新step描述 + 预计5分钟
→ Step卡片底部显示提示文字: "已切换到低能量模式 · 轻量但有进展"
```

## 九、中断检测（Recovery Pattern）

```python
# session_service.py — 每次App打开时调用
def start_new_session() -> str:  # 返回 trigger
    last = state_repo.get_last_session()
    if last and last.ended_at:
        gap_hours = (now_ms() - last.ended_at) / 3_600_000
        if gap_hours > config.SESSION_GAP_THRESHOLD:   # 默认 2.0
            return 'resume'   # → Agent 使用 Recovery pattern
    return 'auto'
```

Recovery pattern Prompt 要求：
- 生成**低门槛再入口**Step
- **不引用**上次未完成的内容
- 优先选择 `in_progress` 块中最近有过进展的那个
